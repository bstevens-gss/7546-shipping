## 4. Section Templates by Classification Type

Each template below defines the required and optional sections for a quote of that type. Sections are numbered sequentially starting at 1.

**NOTE (2026-04-02):** Standard spec generation now produces a single 3-section ServiceWeb Quote (see Section 9 in AGENTS.QUOTING.md). The full-agreement templates below are retained as **legacy reference** for custom deliverables. Components marked **"Legacy — full agreement only"** (stat-grid, phase-card, flow-container, budget-bar, signature-grid, collapsible sections, toggle script) are not used in standard ServiceWeb spec generation.

**Collapsibility Rules (Legacy):** In the full agreement format, Type A/B/D sections were collapsible (Terms & Conditions started collapsed). Type C sections were not collapsible. This does not apply to the ServiceWeb format.

### 4.1 Type A -- DaaS (Discovery / Undefined Scope)

Use when the project is 100+ hours and scope is not yet fully defined. The customer funds a block of hours for iterative discovery and development.

**Required sections:**

| # | Section | Components Used | Content Guidelines |
|---|---------|----------------|-------------------|
| 1 | Solution | p, ul, stat-grid | Problem statement describing the current state. Proposed solution (screens/tools). Stat grid: engagement model, initial hours, hourly rate, review cadence. |
| 2 | Scope Direction | h4 per step, p, info-box | Step-by-step description of the planned work, organized by module. Close with an info-box explaining this is "initial direction" that will be refined iteratively. |
| 3 | Mockup Screenshots | mockup-gallery | **Mandatory.** Build a MOCKUP `.g2u` (see Section 9.3 in agents/AGENTS.QUOTING.md), screenshot it, and embed here. Group by module under h4 headings. |
| 4 | Integration Architecture | architecture-diagram, data-table, warning-box | System context diagram. Integration points table (Module, Direction, Points). Key business rules in a warning-box. |
| 5 | DaaS Engagement Provisions | resp-grid | Customer responsibilities (numbered, 8-10 items). Developer responsibilities (numbered, 4-6 items). |
| 6 | Good Faith Estimate | stat-grid, info-box | Stat grid: hours, rate, total. Info-box: billing frequency and transparency notes. |
| 7 | Terms &amp; Conditions | ol | Numbered terms covering: dedicated developer, iteration review, scope changes, billing frequency, cancellation, 30-day feedback window, ServiceWeb coordination. |
| 8 | Authorization | stat-grid, signature-grid, info-box | Repeat financial stat grid. Signature blocks. Reference documents info-box. |

**Classification badge text:** `TYPE A — DaaS (100+ Hours)`

### 4.2 Type A -- Block-Hour (Defined Scope)

Use when the project is 100+ hours with a defined (though potentially evolving) scope. Work is billed against incremental hour blocks.

**Required sections:**

| # | Section | Components Used | Content Guidelines |
|---|---------|----------------|-------------------|
| 1 | Executive Summary | p, ul, stat-grid, info-box | Problem + solution narrative. Stat grid: screen count, steps, grids, initial block hours. Info-box: billing model explanation. Second info-box: "About This Spec" caveat. |
| 2 | Scope of Work | h3 per module, h4 per step/tab, p, ul | Detailed scope organized by module (2.1, 2.2, etc.). Include subsections for integration points, custom tables, data sources if applicable. |
| 3 | Mockup Screenshots | mockup-gallery | **Mandatory.** Build a MOCKUP `.g2u` (see Section 9.3 in agents/AGENTS.QUOTING.md), screenshot it, and embed here. Group by module under h4 headings. Include alt text and descriptive captions. |
| 4 | Deliverables | data-table | Table columns: #, Deliverable, Type, Description. List every concrete deliverable. |
| 5 | Integration Architecture | architecture-diagram, data-table, warning-box, info-box | System context diagram. Integration points table. Data flow diagram. Custom tables table. Business rules warning-box. Future opportunities info-box. Part numbering strategy (if applicable). |
| 6 | Phases of Work | phase-grid, info-box | Phase cards (typically 4). Info-box explaining agile block-hour approach. |
| 7 | Billing &amp; Payment Terms | data-table, ul, flow-container, info-box | Rate table (standard + overtime). Block-hour model details as ul. Billing workflow flow diagram. "No Surprise Billing" info-box. |
| 8 | Budget Management &amp; Alerts | data-table, budget-visual | Threshold table (70%, 90%, 100% with actions). Budget bar visual. |
| 9 | Change Order Process | h3, ul, flow-container | What requires a CO. What does NOT. CO approval flow diagram. |
| 10 | Assumptions &amp; Prerequisites | ul, checklist, warning-box | Assumptions as standard ul. Prerequisites as checklist. Delay warning-box. |
| 11 | Testing Plan | h3, test-case divs | One test case per scope item. Each test case MUST include Objective, Steps, and Expected Results (all `<ul><li>` bullets). Use inline-style test-case pattern from Section 5.9 in AGENTS.QUOTING.md. Number sequentially: TC-01, TC-02, etc. |
| 12 | Terms &amp; Conditions | h3, ol | **Legacy — full agreement only.** Subsections: Engagement Terms, IP, Warranty & Support, Cancellation, Limitation of Liability. |
| 13 | Authorization | stat-grid, signature-grid, info-box | **Legacy — full agreement only.** Repeat financial stat grid. Signature blocks. Reference documents info-box. |

**Classification badge text:** `TYPE A — BLOCK-HOUR (100+ Hours)`

### 4.3 Type B -- Time & Materials (20-100 Hours)

Use for medium-complexity projects with a defined scope. Work is billed weekly or bi-weekly against an estimated budget with a 15-20% buffer.

**Required sections:**

| # | Section | Components Used | Content Guidelines |
|---|---------|----------------|-------------------|
| 1 | Executive Summary | p, ul, stat-grid, info-box | Problem + solution. Stat grid: deliverable count, estimated hours, hourly rate, billing frequency. Info-box: T&M billing model explanation. |
| 2 | Scope of Work | h3, h4, p, ul | Detailed description of each deliverable. Less modular than Type A -- typically one main screen or feature. |
| 3 | Mockup Screenshots | mockup-gallery | **Mandatory.** Build a MOCKUP `.g2u` (see Section 9.3 in agents/AGENTS.QUOTING.md), screenshot it, and embed here. |
| 4 | Deliverables | data-table | Table columns: #, Deliverable, Type, Description. |
| 5 | Timeline &amp; Milestones | phase-grid or data-table | Use phase cards for 2-4 phases, or a data-table for milestone-based timelines. |
| 6 | Billing &amp; Payment Terms | data-table, ul, info-box | Rate table. T&M billing details: estimated hours, buffer percentage, billing frequency. Info-box: budget tracking commitment. |
| 7 | Budget Management | data-table, budget-visual | Threshold table (e.g., 80%, 100%, 115% of estimate). Budget bar if helpful. |
| 8 | Change Order Process | ul, flow-container | Abbreviated: what triggers a CO, approval flow. |
| 9 | Assumptions &amp; Prerequisites | ul, checklist | Assumptions. Prerequisites checklist. |
| 10 | Testing Plan | h3, test-case divs | One test case per scope item. Each test case MUST include Objective, Steps, and Expected Results (all `<ul><li>` bullets). Use inline-style test-case pattern from Section 5.9 in AGENTS.QUOTING.md. Number sequentially: TC-01, TC-02, etc. |
| 11 | Terms &amp; Conditions | ol | **Legacy — full agreement only.** Abbreviated terms covering: billing, IP, warranty (30 days), cancellation. |
| 12 | Authorization | stat-grid, signature-grid | **Legacy — full agreement only.** Financial summary stat grid. Signature blocks. |

**Classification badge text:** `TYPE B — TIME & MATERIALS (20-100 Hours)`

### 4.4 Type C -- Quick Fix (2-20 Hours)

Use for small, well-defined tasks: bug fixes, minor modifications, report tweaks, quick features. Billed automatically on a weekly cycle.

**Required sections:**

| # | Section | Components Used | Content Guidelines |
|---|---------|----------------|-------------------|
| 1 | Summary | p, stat-grid | Concise problem statement (1-2 paragraphs). Stat grid: estimated hours, hourly rate, buffer, total estimate. |
| 2 | Scope | p, ul | Specific deliverables. What is included and what is excluded. |
| 3 | Estimate | data-table | Table: Task, Estimated Hours, Rate, Amount. Include buffer row (10-15%) and grand total row. |
| 4 | Assumptions | ul | Brief list of assumptions and prerequisites. |
| 5 | Terms | info-box, ol | Info-box: auto-billing weekly, no customer sign-off required (per PO terms). Abbreviated terms: billing increment, warranty (30 days), cancellation. |
| 6 | Authorization | stat-grid, signature-grid | Financial summary. Signature blocks (can omit PO line if PO is already on file). |

**Classification badge text:** `TYPE C — QUICK FIX (2-20 Hours)`

**Type C specifics:**
- Sections should NOT be collapsible -- the document is short enough to read straight through. Use standard `<div>` containers with the same margin/padding styling instead of the toggle pattern.
- No phase cards, no budget bar, no change order section.
- Stat boxes can use a smaller grid: `minmax(120px, 1fr)`.

### 4.5 Type D -- Support/Retainer (Ongoing)

Use for ongoing support contracts and retainer agreements with monthly hour allocations.

**Required sections:**

| # | Section | Components Used | Content Guidelines |
|---|---------|----------------|-------------------|
| 1 | Engagement Summary | p, stat-grid | Overview of the support relationship. Stat grid: monthly hours, hourly rate, monthly retainer amount, response SLA. |
| 2 | Services Covered | data-table or ul (two columns) | What IS covered (standard support, bug fixes, minor mods, configuration). What is NOT covered (new development, major enhancements -- these become separate Type A/B/C quotes). |
| 3 | Rate &amp; Hours | data-table, info-box | Rate table (standard, overtime, emergency). Monthly allocation details. Info-box: rollover/expiration policy for unused hours. |
| 4 | Response Times &amp; SLA | data-table | Priority levels: Critical, High, Normal, Low. Response time and resolution target for each. |
| 5 | Billing | ul, info-box | Monthly invoicing schedule. Overage billing policy. Pre-approval requirements for hours beyond allocation. |
| 6 | Terms | ol | Duration (typically 12 months), auto-renewal, cancellation notice period (30/60/90 days), rate adjustment schedule. Start collapsed. |
| 7 | Authorization | stat-grid, signature-grid | Financial summary. Signature blocks. |

**Classification badge text:** `TYPE D — SUPPORT/RETAINER (Ongoing)`

## 6. JavaScript (Legacy — Full Agreement Only)

**Not used in standard ServiceWeb spec generation.** Retained for custom full-agreement deliverables.

Every full-agreement document includes this script block at the end of the body, before `</body>`:

```html
<script>
    function toggleSection(header) {
        var section = header.parentElement;
        section.classList.toggle('collapsed');
        header.classList.toggle('collapsed');
    }
</script>
```

No external dependencies. No CDN imports. Pure vanilla JS.

For Type C documents, omit this script entirely since sections are not collapsible.

## 7. Document Skeleton (Legacy — Full Agreement Only)

**Not used in standard ServiceWeb spec generation.** The ServiceWeb HTML skeleton is in [SERVICEWEB.md Section 12.4](SERVICEWEB.md). This skeleton is retained for custom full-agreement deliverables.

Every full-agreement document follows this HTML skeleton. Fill in the CSS from the component library above, then populate sections per the classification template.

```html
<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>{Customer} - {Project Name} | {Agreement Type} Agreement</title>
    <style>
        /* Reset */
        * { margin: 0; padding: 0; box-sizing: border-box; }

        /* Paste ALL CSS from components used by this document type */
        /* Body, container, header, doc-meta, classification-badge */
        /* Sections, info-grid, info-card, stat-grid, stat-box */
        /* Callout boxes, data-table, etc. */
        /* Responsive and print media queries LAST */
    </style>
</head>
<body>

<div class="container">

    <!-- HEADER -->
    <header>
        <h1>{Project Name}</h1>
        <p class="subtitle">{Agreement Type} Agreement</p>
        <div class="doc-meta">
            <span>Customer: {Customer Name}</span>
            <span>Project Code: {Code}</span>
            <span>Date: {Date}</span>
            <span>Version: {Version}</span>
        </div>
        <div class="classification-badge">TYPE {X} &mdash; {MODEL} ({Hours})</div>
    </header>

    <!-- INFO GRID -->
    <div class="info-grid">
        <!-- Customer info card -->
        <!-- Project/engagement info card -->
    </div>

    <!-- SECTIONS (per classification template) -->
    <!-- ... -->

</div>

<script>
    function toggleSection(header) {
        var section = header.parentElement;
        section.classList.toggle('collapsed');
        header.classList.toggle('collapsed');
    }
</script>

</body>
</html>
```

## 8. CSS Assembly Rules (Legacy — Full Agreement Only)

**Not used in standard ServiceWeb spec generation.** The ServiceWeb format uses inline styles per Section 5.10 in AGENTS.QUOTING.md. These rules are retained for custom full-agreement deliverables.

When generating a full-agreement document, include ONLY the CSS classes that are actually used by that document type. Do not include unused component styles.

**Type A (Block-Hour) uses ALL components.** Include the full CSS.

**Type A (DaaS) uses:** header, doc-meta, classification-badge, info-grid, info-card, section, section-header, section-content, stat-grid, stat-box, info-box, warning-box, data-table, resp-grid, resp-card, mockup-gallery, mockup-item, signature-grid, signature-block, sig-line, sig-label, architecture diagrams (inline styles). Does NOT use: phase-grid, phase-card, flow-container, flow-step, flow-arrow, budget-visual, budget-bar, checklist, success-box, type-badge.

**Type B uses:** header, doc-meta, classification-badge, info-grid, info-card, section, section-header, section-content, stat-grid, stat-box, info-box, warning-box, success-box, data-table (with total/buffer/grand-total rows), phase-grid, phase-card (or data-table for milestones), flow-container, flow-step, flow-arrow, checklist, mockup-gallery, mockup-item, signature-grid, signature-block, sig-line, sig-label, budget-visual, budget-bar. May NOT need: resp-grid, architecture diagrams.

**Type C uses (minimal set):** header, doc-meta, classification-badge, info-grid, info-card, stat-grid, stat-box, info-box, data-table (with buffer and grand-total rows), signature-grid, signature-block, sig-line, sig-label. Does NOT use: section collapse mechanism, phase-grid, flow-container, resp-grid, mockup-gallery, budget-visual, checklist, warning-box, success-box.

**Type D uses:** header, doc-meta, classification-badge, info-grid, info-card, section, section-header, section-content, stat-grid, stat-box, info-box, warning-box, data-table, signature-grid, signature-block, sig-line, sig-label. May NOT need: phase-grid, flow-container, resp-grid, mockup-gallery, budget-visual, checklist, success-box.
