## 12. ServiceWeb Quote Format

The ServiceWeb Quote is the **single spec document** generated for every project. The primary output format is **DOCX** (Word), generated by `scripts/generate_quote_docx.py`. It matches the GSS ServiceWeb Quote Details page layout and is structured as three clearly separated sections for easy copy-paste into ServiceWeb fields.

**Reference artifact:** `ServiceWeb - Quote Details (15890-0).html` (saved page from actual ServiceWeb for SPS)

### 12.1 ServiceWeb Page Structure

The ServiceWeb quote page has these fixed zones (in order). The generated HTML mirrors this layout with three content sections instead of two:

| Zone | Type | What Goes Here |
|------|------|---------------|
| Header | Auto-generated | "Quote for {CUSTOMER NAME}" |
| Metadata row | Auto-generated | Quote-Rev, Expiration Date, Contact, Call Reference |
| Title | Single text field | Project title (one line) |
| **Section 1: Description** | Rich text editor | Executive summary, assumptions (with existing-project disclaimer if modification), prerequisites |
| **Section 2: Specifications (Customer Facing)** | Rich text editor (first half) | User-friendly scope of work (no code refs), deliverables table, full Testing Plan (all test cases) |
| **Section 3: Technical Specifications** | Rich text editor (second half) | Developer-detail scope (code refs, SQL, hooks), architecture diagram, integration points table, callout boxes |
| Quote Details | Line-item table | Sequence, Type (GAB/Config/etc.), Hours, Rate, Total, Description |
| Attachments | File upload list | RFQ documents, supporting files |
| Disclaimers | Auto-generated | Standard 6-item GSS boilerplate (do NOT modify) |
| Signature | Auto-generated | Single signature line + date |

**Paste workflow:** Section 1 is copied into the ServiceWeb "Description" field. Sections 2 and 3 are copied sequentially into the ServiceWeb "Specifications" field (Section 2 first, then Section 3 below it). The sections are separated by a subtle thin gray line in the HTML preview.

### 12.2 3-Section Content Routing

| Section | Pastes Into | Content |
|---------|-------------|---------|
| **1. Description** | ServiceWeb "Description" field | Executive summary paragraphs, Assumptions h3 (with existing-project disclaimer as first item if modification per Section 5.12), Prerequisites h3 |
| **2. Specifications (Customer Facing)** | ServiceWeb "Specifications" field (first half) | "Scope of Work" h3 with numbered list in plain language (no `<code>` refs, no subroutine names), "Deliverables" h3 with table, "Testing Plan" h3 with all test cases (TC-01, TC-02, ...) per Section 5.9 |
| **3. Technical Specifications** | ServiceWeb "Specifications" field (second half) | "Technical Scope of Work" h3 with full developer detail (`<code>` refs, SQL, hooks, SetColumnProperty), "Architecture Overview" h3 with diagram, "Integration Points" h3 with table, callout boxes for schema discovery / conflict risks |

**What is NOT included** (retired from standard spec generation):
- Collapsible sections / toggle mechanism
- Stat grid boxes, phase cards, flow diagrams
- Budget bar / threshold alerts, change order process
- Billing & payment terms, terms & conditions (GSS disclaimers cover these)
- Dual-column signature blocks (ServiceWeb provides its own)

### 12.3 Standard GSS Disclaimers

These are auto-generated by ServiceWeb. Include them verbatim in the companion HTML for preview accuracy:

1. **GSS Version Requirement:** GSS version 2020.1 or higher required.
2. **Purchase Order and Billing:** PO required in full before commencement. Monthly billing for multi-month projects. Additional POs for overages.
3. **Acceptance of Project Scope/Quotes and Change Requests:** Acceptance confirms review. Change requests subject to requote process.
4. **Project Cancellation:** Customer may cancel; costs to date are billed.
5. **Training:** Not included unless stated. Available as separate line item or virtual session.
6. **Terms and Conditions/Service:** Serviced per Software Licensing Agreement. Projects are property of Global Shop Solutions.

### 12.4 DOCX Generation

The standard output format is **DOCX** (Microsoft Word), generated by a Python script. The AI agent produces structured JSON content; the script converts it into a professionally styled Word document.

**Script:** `scripts/generate_quote_docx.py`
**Dependencies:** python-docx 1.2.0, Pillow 12.0.0 (both pre-installed)

**Usage:**
```
python scripts/generate_quote_docx.py <content.json> "<output.docx>"
```

**Example:**
```
python scripts/generate_quote_docx.py adams_eft_content.json "ADAMS - EFT Export ATB Bank ServiceWeb Quote.docx"
```

**JSON Content Schema:** Documented as a docstring in the script. The agent builds a JSON file following this schema:

- `meta` — customer name, abbreviation, project title, quote-rev, expiration, contact, call reference, classification, modification flags
- `section1_description` — executive_summary (paragraphs), assumptions (bullets), prerequisites (bullets)
- `section2_specifications` — scope_items, deliverables (with `type` column), test_cases (with optional `reference` and sub-headed `expected_results`)
- `section3_technical` — content_blocks (ordered array of `heading`, `sub_heading`, `table`, `code_block`, `callout`), architecture (flexible box diagram), integration_points, screenshots
- `quote_details` — line_items (with `is_buffer` flag), total_hours, total_amount

**Inline formatting:** Use lightweight markdown in JSON string values: `**bold**`, `*italic*`, `` `code` ``. The script parses these into Word text runs.

**File naming:** `{Customer Abbrev} - {Project Name} ServiceWeb Quote.docx`

### 12.5 Styling Rules for ServiceWeb Paste

When content from the DOCX is copied from Word and pasted into the ServiceWeb rich text editor:

- **Word clipboard format** puts both RTF and HTML on the clipboard. DevExpress rich text editors handle Word paste with high formatting fidelity -- tables, bold/italic, font colors, and backgrounds generally survive intact.
- **Explicit colors are preserved:** python-docx sets `RGBColor` on every text run, so Word always exports explicit colors in the clipboard HTML. This is inherently more reliable than browser-based copy-paste.
- **Tables paste well:** Word table formatting (borders, shading, cell padding) transfers cleanly to DevExpress editors.

**ServiceWeb paste dark mode still applies:** While Word's clipboard format is more resilient than browser HTML, the DevExpress rich text editor can still override inherited colors in dark mode. The DOCX generator addresses this by setting explicit `font.color.rgb` on every text run. See Section 5.10.2 in [agents/AGENTS.QUOTING.md](../AGENTS.QUOTING.md) for the full rule.

**No browser dark mode concern:** DOCX files are opened in Word, not a browser. Browser dark mode (Section 5.10.1) is irrelevant for DOCX output.
