# GAB Syntax Linter

Static analysis for `.g2u` and `.lib` files. Catches invented APIs, wrong syntax
patterns, and convention violations before runtime.

## Quick Start

```powershell
$lint = Join-Path $env:USERPROFILE ".cursor/skills/gab-lint/scripts/gab-lint.ps1"
powershell -NoProfile -ExecutionPolicy Bypass -File $lint -Path "<file.g2u>"
```

Multiple files or a directory:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File $lint -Path "path/to/folder" -Recurse
```

## Parameters

| Parameter | Required | Default | Description |
|-----------|----------|---------|-------------|
| `-Path`   | Yes      | --      | File or directory to lint |
| `-Recurse`| No       | false   | Recurse into subdirectories |
| `-Severity`| No      | `error` | Minimum severity: `error`, `warning`, or `convention` |
| `-Format` | No       | `text`  | Output format: `text` or `json` |

## Exit Codes

| Code | Meaning |
|------|---------|
| 0    | No issues found at or above the requested severity |
| 1    | Issues found |
| 2    | File not found or invalid arguments |

## Rule Categories

| Category | ID Prefix | Scope |
|----------|-----------|-------|
| Syntax & Operators | A | Math ops, string concat, String.Build |
| Properties & Accessors | B | FieldVal chaining, UBound, PSQLFriendly |
| GUI Read/Write | C | V.Screen vs Gui, form types, control props |
| Grid & Data Binding | D | DataSource vs MainView, child DT naming |
| DataTable & ODBC | E | Scope, connection naming, hardcoded DSN |
| File I/O | F | Argument order, invented functions |
| CallSub & Error Handling | G | Array args, nested Try, missing error handler |
| Invented APIs | H | Functions/methods that do not exist in GAB |

## Agent Workflow

When linting a GAB file:

1. Run the linter script against the target file
2. Review the output for errors and warnings
3. Fix issues in order: errors first, then warnings
4. Re-run the linter to verify fixes
5. Only then proceed to sign/run the script

## Suppressing Rules

Add a comment on the line before the flagged code:

```gab
'-- gab-lint-ignore: B1
V.Local.s.Set(V.Screen.frm!txt.Text())
```

## Allowlist API Validation (R-rules)

In addition to the pattern-based rules above, gab-lint now supports **allowlist-based
API validation** powered by a registry extracted directly from the OCTSRS runtime source
code. This catches ANY invented/hallucinated API call -- not just previously-documented
mistakes.

```powershell
$validate = Join-Path $env:USERPROFILE ".gab-agents/tools/Validate-GabApi.ps1"
powershell -NoProfile -ExecutionPolicy Bypass -File $validate -Path "<file.g2u>"
```

| Rule | Severity | Catches |
|------|----------|---------|
| R1   | ERROR    | Unknown namespace (F.xxx not valid) |
| R2   | ERROR    | Unknown nominative (F.Intrinsic.xxx not valid) |
| R3   | ERROR    | Invalid method (will cause Error 999000 at runtime) |
| R4   | ERROR    | Invalid V.Ambient.xxx variable |
| R5   | ERROR    | Invalid V.Caller.xxx variable |

The registry is generated by `Extract-GabApiRegistry.ps1` from the OCTSRS source code
and contains **1,683 valid methods**, **120 V.Ambient variables**, and **46 V.Caller
variables**.

## Agent Workflow (Updated)

When linting a GAB file:

1. Run the pattern-based linter (`gab-lint.ps1`) against the target file
2. Run the allowlist validator (`Validate-GabApi.ps1`) against the same file
3. **NEW -- MCP tiebreaker**: If the allowlist flags a command AND MCP Intelligence is connected, call `validate_gab_command` via `gab-commands` proxy as a tiebreaker. MCP has 5,421 definitions (vs 1,683 in local registry). **MCP wins if it says valid.**
4. Review ALL output for errors and warnings
5. Fix issues in order: errors first, then warnings
6. Re-run both checks to verify fixes
7. Only then proceed to sign/run the script

### MCP Validation Step (when connected)

```
CallMcpTool(
  server: "user-mcp-intelligence",
  toolName: "call_proxy_tool",
  arguments: { "server": "gab-commands", "tool_name": "validate_gab_command", "arguments": "{\"commandName\": \"Function.ODBC.Connection!*con.Execute\"}" }
)
```

**IMPORTANT**: For ODBC instance-based commands, use the wildcard form (`!*con`) not the literal instance name.

## Known False Positives (D12/D13)

Rules D12 and D13 in the VS Code extension flag ODBC usage as "deprecated." This is **incorrect** -- ODBC is fully supported (41% of production scripts use it, confirmed via OCTSRS source and MCP validation).

- `gab-lint.ps1` has downgraded these to `convention` severity with corrected messaging
- The VS Code extension still shows warnings until a rebuild is released (extension v2.2.0 pending)
- **Suppress per-line**: `'-- gab-lint-ignore: D12, D13`
- **Do NOT "fix" ODBC code because lint flagged it** -- ODBC is valid and actively used

## Skill Coherence: gab-lint + gab-test-debug

These skills work together. Key interactions:

- `gab-test-debug` instructs you to use `F.Intrinsic.Debug.SetLA`, `EnableLogging`, `DumpVariableList` -- lint will NOT flag these (they are valid automation-safe methods)
- `gab-test-debug` instructs you to use ODBC connections for data validation queries -- lint D12/D13 are convention-level only, NOT errors
- **NEVER use `F.Intrinsic.Debug.SetLevel`** during automated test-debug -- it blocks with a dialog. Lint D17 flags `Debug.Resume` (error), D18-D20 warn on interactive-only methods (ShowCallerInfo, Stop, InvokeDebugger)
- When lint flags something that test-debug told you to use, **trust the skill guidance over the lint warning**
